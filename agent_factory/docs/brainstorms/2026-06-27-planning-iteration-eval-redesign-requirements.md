# Planning Iteration Eval Redesign

## Summary

Redesign the factory's planning eval from "reproduce a golden plan / measure coverage" to two
questions: did the plan capture every **explicitly-requested** feature, and did it **surface** (include,
or flag as a forced decision) the **implied** features and decisions needed to build them. The implied
set is generated by the Praxis planning checks — the same checks `af-intake` enforces — so one set
of facts drives both enforcement and measurement, and the result is read as a directional signal, not a
single-run pass/fail.

## Problem Frame

The current `plan_repro` eval scores feature coverage of a proxy planner against a golden that is itself
the output of one human-refined planning pass. Two failures fall out of that framing:

- **It measures the wrong class of failure.** The forgot-password break was not a missing feature —
  password-reset *was* planned. It was a missing *decision*: no email transport was ever chosen, so the
  reset link could not be delivered. Coverage-vs-golden is structurally blind to a feature that is
  present but under-specified.
- **It optimizes noise.** Hole counts bounced 12 → 8 → 4 → 6 across rounds; the levers (lenses, a gate
  pass) are swamped by single-shot LLM variance and an adversarial refuter that rejects legitimate-but-
  reworded coverage. Tuning toward a green number over-fits that noise, and the real factory is
  human-gated and multi-pass, so a one-shot proxy's count is not even representative.

The redesign separates the two qualities a plan must have (breadth and depth) and measures depth — which
nothing currently does — using the checks the factory already enforces.

## Key Decisions

- **Two questions, not one coverage score.** Breadth (every explicit feature present) and depth (every
  present feature fully resolved) are different qualities and are scored separately.
- **Explicit ground truth comes from the raw PRD, not a golden plan.** The explicit-feature list is
  extracted from what was literally asked for, which removes the circularity of grading a plan against
  another plan.
- **The implied set is generated, not authored.** Applying each Praxis planning check to each explicit
  feature yields the implied needs to score against — so the eval auto-grows with the checks and never
  needs a separate answer key.
- **One source of truth for enforcement and measurement.** The Praxis planning checks
  (`scope="planning"`) that `af-intake` enforces are the same lenses the eval uses. Adding a check
  once arms both sides; a weak check weakens both, by design.
- **The result is a directional signal.** Persistent holes across runs are the signal; a single run is
  not a pass/fail gate. Reliability stays with the human-gated factory, which the eval informs.

Source-of-truth fan-out (the coupling that makes the loop compound):

```
                         Praxis planning checks  (scope="planning")
                          e.g. external-service-provider-decision
                                    |
                 +------------------+------------------+
                 |                                     |
       af-intake ENFORCES                eval GENERATES implied set
       (forces the decision when              (lens x explicit feature ->
        forming a plan)                        implied need -> score surfacing)

       add ONE check  ->  factory forces it  AND  eval measures it
```

## Requirements

**Eval questions**

R1. The planning eval scores two distinct questions rather than one coverage number: explicit coverage
(R2) and implied-need surfacing (R3).

R2. Explicit coverage — every feature explicitly requested in the source PRD must appear in the plan.
The ground truth is the explicit-feature list extracted from the raw PRD, not a refined golden plan.

R3. Implied-need surfacing — for each generated implied need (R4), the plan must surface it; an
un-surfaced implied need is a depth hole.

**Implied-set generation**

R4. The implied set is generated, not hand-authored: each Praxis planning check (lens) applied to each
explicit feature yields the implied needs the eval scores against.

R5. Implied needs include implied features (an email transport for "sends a reset email") and implied
behaviors, edges, and decisions (reset token is single-use; an expired token is rejected) — both produced
by the applicable lenses.

R6. A need that no lens generates is outside the eval's view by design. Discovering such a gap is the
trigger to add a new planning check (the compounding loop), not to hand-patch the eval.

**Surfacing definition**

R7. "Surfaced" is satisfied either way: the plan includes the implied need as a requirement/decision, OR
the plan flags it as an explicit forced decision (raised to the user, or auto-suggested with a default).
Silence on an implied need is the only failure.

**Shared source of truth**

R8. The Praxis planning checks (`scope="planning"`) are the single source of truth — the checks
`af-intake` enforces during planning are the same ones the eval uses to generate the implied set.
There is no separate eval answer key.

R9. Adding one planning check makes the factory force it and the eval measure it; a missing or weak check
weakens both. This coupling is intended and must be preserved, not worked around with eval-only lists.

**Depth floor**

R10. "Fully defined" terminates at: every decision answered AND the feature's behaviors and edges
specified — bounded by the lenses that generate them, so depth-checking is finite rather than an
open-ended demand for more detail.

**Signal interpretation**

R11. The eval result is a directional / persistent-gap signal, not a single-run pass/fail gate.
Robustness comes from holes that persist across runs, not one run reaching zero.

R12. Planning reliability remains the human-gated factory (`af-intake` plus the human clearing the
gate). The eval informs which lenses to add; it does not replace the gate.

## Acceptance Examples

AE1. **Covers R3, R7 — the SMTP failure.** Explicit feature "a user receives an emailed reset link." The
external-service lens generates the implied need "email transport decision." PASS if the plan names a
transport (e.g. SES) OR flags "email transport — undecided, choose a provider." FAIL if the plan is
silent on the transport (the original break).

AE2. **Covers R7 — flagged-decision counts.** For auth credential email, a plan that surfaces the
managed-vs-custom fork (AWS Cognito's built-in email vs custom auth + standalone SMTP) as an open
decision PASSES even if unresolved — surfacing the fork is the bar, not resolving it.

AE3. **Covers R2 — explicit coverage is independent.** A feature on the PRD's explicit list that is
absent from the plan fails explicit coverage regardless of how well implied needs were surfaced.

## Scope Boundaries

- Harness implementation — how explicit features are extracted, how "surfaced" is detected in planner
  output, and the scoring math — is planning's job, not this brief.
- The `build_repro` (build-loop) eval is a separate track.
- The factory's human-gated planning structure is unchanged; this redesigns the eval, not the gate.
- The golden is not eliminated, only shrunk: the explicit-feature list is still a (smaller) ground truth
  that must be maintained per project.

## Dependencies / Assumptions

- Praxis planning checks exist and are queryable by `scope="planning"` (currently two:
  `external-service-provider-decision`, `password-complexity-policy`).
- The planner's output is machine-parseable for both explicit features and flagged/forced decisions.
- The explicit-feature list can be reliably extracted from the raw PRD (authored once or semi-automated).

## Outstanding Questions

**Resolve before planning**

- How does the planner represent a "surfaced but unresolved" decision so the scorer can detect it — a
  structured `open_decisions` / `forced_decisions` field in the plan output, versus free-text the judge
  must infer? This shapes both the planner output contract and the scorer.

**Deferred to planning**

- Exact extraction of the explicit-feature list from the PRD (authored vs LLM-extracted vs hybrid).
- The scoring model over the (explicit feature × lens) matrix, and how partial surfacing is graded.
- Whether to keep a periodic adversarial pass purely to *discover* new lenses (its finds become checks,
  not scores), per the backstop option raised in dialogue.
