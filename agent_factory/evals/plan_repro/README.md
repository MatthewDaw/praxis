# Planning eval — explicit coverage + implied-need surfacing

Proves the factory's PLANNING process answers two questions for a given PRD:

1. **Explicit coverage** — did it capture every feature the raw PRD EXPLICITLY asked for?
2. **Implied-need surfacing** — did it SURFACE the implied features/decisions needed to build those —
   either by admitting them or by raising/defaulting a forced decision?

The second question is the depth dimension and the reason this eval exists. The archetypal failure: the
PRD says "email a password-reset link" (explicit) but never names an **email transport** (implied) — so
the reset link is never delivered. Coverage-of-features is blind to that; this eval is not.

## The implied set is lens-generated (single source of truth)

The implied needs are NOT a hand-authored answer key. They are GENERATED by applying each **Praxis
planning check** (`scope="planning"`) to the plan. Those checks are the same lenses `af-intake` enforces
when forming a plan — so adding ONE planning check simultaneously makes the factory force it AND makes
this eval measure it. A depth hole the eval finds is a decision `af-intake` should have surfaced. A need
no lens encodes is invisible by design — discovering one is the trigger to add a new planning check (the
compounding loop), not to patch the eval.

## What's under test: the REAL af-intake pipeline (primary)

`run_eval_pipeline.py` drives the **real** `af-intake` forcing — not a proxy:

1. Provision an isolated Praxis space; seed the planning lenses into it.
2. Drive `af-intake` (headless `claude` CLI, subscription, tools enabled) in **UNATTENDED** mode on the
   fixed PRD (`docs/inspiration/`): push in every stated feature, sweep every planning check, and for
   every forced decision the audit raises **choose the low-regret default + record an episode** (never
   ask). Leave requirements + decision episodes in the space (no snapshot).
3. Read the produced plan back (requirements + decision episodes).
4. Score: **Q1** explicit coverage vs the EXPLICIT golden (`derived==False`); **Q2** implied-need
   surfacing via the lenses applied to the produced plan (a need surfaced as a requirement OR a recorded
   forced-decision/default passes; an unsurfaced applicable need is a hole).
5. Tear down.

"Working" = both ~complete: the real pipeline ingested every stated feature AND let every planning insight
expose its implied need. A hole is a genuine `af-intake` defect, not a weak proxy's luck.

    # primary — real pipeline (heavy: a full planning session; needs af-intake loaded + Praxis up)
    python -m evals.plan_repro.run_eval_pipeline

### Prerequisite
The `af-intake` skill must be loaded (reload the plugin after the `factory-`→`af-` rename), and Praxis
must be reachable with the repo `.env` credentials. `PIPELINE_AGENT_TIMEOUT` (default 3600s) bounds the
agent run.

## Fast proxy baseline (optional)

`run_eval_subscription.py` scores a controllable single-shot proxy planner instead of the real pipeline —
cheaper and repeatable, but it has no forcing step, so it always shows depth holes (it measures a raw
planner, not `af-intake`). Use it as a quick directional baseline, not the source of truth. The real
pipeline (`run_eval_pipeline.py`) is what answers "would the factory surface everything."

## Signal, not pass/fail

Treat the result as directional — read the **persistent** holes across runs, not a single run's number
(LLM planning is non-deterministic; the adversarial coverage refuter is strict). A persistent depth hole
means a lens the pipeline reliably fails to act on; a persistent explicit hole means a feature it drops.

## Files
- `run_eval_pipeline.py` — PRIMARY: drive the real af-intake → score the produced plan (Q1 + Q2).
- `run_eval_subscription.py` — optional fast proxy-planner baseline (Q1 + Q2 over a single-shot plan).
- `coverage.py` — the coverage engine (per-part sweep, evidence-required + targeted adversarial) used for
  Q1 explicit coverage.
- `depth.py` — Q2: applies each lens to the plan; classifies each applicable capability surfaced /
  flagged / **missing** (a depth hole).
- `praxis_source.py` — the isolated Praxis-space lifecycle (create → clear → seed lenses → read → teardown).
- `claude_cli.py` — subscription-backed `Complete` (headless `claude`, no API key) for the planner + judges.
- `planning-checklist.yaml` — the lens seed (the version-controlled checks the eval rounds through Praxis;
  the durable home is Praxis `scope="planning"` checks that `af-intake` enforces).
- `team-app/golden-features.yaml` — the coverage target; `derived: false` = the EXPLICIT set for Q1,
  `derived: true` = features the raw PRD never stated (the depth dimension the lenses must expose).
- `llm_evaluator.py`, `coverage`/`depth` tests, etc.
