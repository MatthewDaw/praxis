Praxis MVP

scope: MVP only · date: 2026-06-18 · Fuller design parked in future-work/

The one goal: ship the simplest possible Praxis knowledge loop (ingest → store → read) and an eval harness solid enough that we can fan out and write a pile of evals — most of which we expect to fail. Register them, get a baseline number, and iterate from there. Everything past that is deferred.

Scope

Two things, each at its absolute minimum:

  • The core product — the simplest knowledge distillation/retrieval we can write: a CLAUDE.md knowledge graph, an ingestion tool that rewrites it, and a reader tool that returns it.
  • The eval harness — a fully spelled-out eval-case definition plus a runner that executes a case and records a score, so cases can be authored and baselined en masse.
Explicitly out of scope (parked in future-work): selective/ranked retrieval, real knowledge-graph structures, substrate bake-offs, cold-vs-injected lift, attribution/context-control metrics, poison-resistance, per-case ε tuning, the diff-scoped selector, the git-committed scoreboard with verify/compare, and the GitLab CI gate. None of it blocks writing and baselining evals.

Part 1 — Core product (simplest knowledge loop)

Strict OOP: each of the three core pieces is an abstract parent that pins down the function names and their typed inputs/outputs. Concrete variants subclass the parent and solve the same problem differently (MVP ships one trivial variant of each). What must be frozen now is the parent contracts below — variants are free to churn.

Core contracts (freeze these now)

Knowledge graph — the storage primitive both other pieces depend on. MVP variant is a CLAUDE.md file.

# praxis/knowledge/knowledge_graph/parent_knowledge_graph.py
from abc import ABC, abstractmethod

class KnowledgeGraph(ABC):
    """The knowledge store. The 'graph' is whatever a variant persists."""

    @abstractmethod
    def read(self) -> str:
        """Return the full current knowledge content."""

    @abstractmethod
    def write(self, content: str) -> None:
        """Replace the stored knowledge with `content`."""

# variant: knowledge_graph_variants/claude_md_graph.py
#   class ClaudeMdGraph(KnowledgeGraph): def __init__(self, path: Path): ...
#   read() returns the file text; write() overwrites it.

Ingestor — folds an arbitrary insight string into the graph. MVP variant = hardcoded update prompt + one Claude call.

# praxis/knowledge/injestion/parent_injestor.py
from abc import ABC, abstractmethod
from praxis.knowledge.knowledge_graph.parent_knowledge_graph import KnowledgeGraph

class Ingestor(ABC):
    """Distills an insight into the knowledge graph."""

    def __init__(self, graph: KnowledgeGraph) -> None:
        self.graph = graph

    @abstractmethod
    def ingest(self, insight: str) -> str:
        """Fold `insight` into the graph, persist it via self.graph.write(),
        and return the new graph content."""

# variant: injestor_variants/prompt_injestor.py
#   class PromptIngestor(Ingestor):
#       def __init__(self, graph, llm): ...
#       ingest(insight): new = llm(UPDATE_PROMPT, self.graph.read(), insight)
#                        self.graph.write(new); return new
#   UPDATE_PROMPT is a single hardcoded constant for MVP.

Graph reader — retrieves knowledge for the agent, given current context. Signature takes context now even though the MVP variant ignores it and returns everything.

# praxis/knowledge/graph_reader/parent_graph_reader.py
from abc import ABC, abstractmethod
from praxis.knowledge.knowledge_graph.parent_knowledge_graph import KnowledgeGraph

class GraphReader(ABC):
    """Retrieves knowledge for the agent, given current context."""

    def __init__(self, graph: KnowledgeGraph) -> None:
        self.graph = graph

    @abstractmethod
    def read(self, context: str | None = None) -> str:
        """Return knowledge relevant to `context`."""

# variant: grapher_reader_variants/whole_file_reader.py
#   class WholeFileReader(GraphReader):
#       def read(self, context=None) -> str: return self.graph.read()  # context ignored
#   A thin adapter registers read() as a Claude tool; the method stays pure.

M1. KnowledgeGraph parent + ClaudeMdGraph variant product

MVP done when: ClaudeMdGraph(path).read()/.write() round-trips a CLAUDE.md file. The "graph" is just the file text — no structure, no index.

M2. Ingestor parent + PromptIngestor variant product

MVP done when: PromptIngestor(graph, llm).ingest("some insight") feeds the hardcoded UPDATE_PROMPT + current graph + insight to Claude, writes the result back, and returns the new content.

M3. GraphReader parent + WholeFileReader variant product

MVP done when: WholeFileReader(graph).read() returns the full graph, and a thin adapter exposes it as a registered Claude tool the agent under eval can call.

Part 2 — Eval harness

The eval-case definition (spelled out in full)

An EvalCase is exactly these fields — nothing more for MVP:

FieldMeaning
deterministic_checksPython code that runs against the produced output and returns pass/fail (e.g. "tests pass", "function X exists", "string Y present").
rubricA checklist scored by an LLM judge for the qualitative things deterministic checks can't capture.
target_commitThe git commit representing the desired end state — the reference the checks/rubric are authored against.
start_commit optionalThe commit the agent starts from. Optional — absent means start from an empty/clean baseline.
seed_promptThe instruction handed to the agent (e.g. "build this feature").

The schema — eval_def.py (Pydantic, frozen now)

Cases are authored as data (YAML undecided but likely) and load into this model. Deterministic checks are references to callables under evals/deterministic_checks/; the rubric is inline for MVP.

# praxis/knowledge/evals/eval_def.py
from pydantic import BaseModel, Field, model_validator

class DeterministicCheckRef(BaseModel):
    """Points at a registered callable in evals/deterministic_checks/."""
    name: str
    ref: str                      # dotted path, e.g. "deterministic_checks.builds:tests_pass"
    params: dict = Field(default_factory=dict)

class RubricItem(BaseModel):
    id: str
    criterion: str                # what the LLM judge evaluates
    weight: float = 1.0

class Rubric(BaseModel):
    id: str
    items: list[RubricItem]

class EvalCase(BaseModel):
    id: str
    seed_prompt: str                          # instruction handed to the agent
    target_commit: str                        # desired end state / reference
    start_commit: str | None = None           # optional; None => clean baseline
    repo: str | None = None                   # where the commits live (defaults to this repo)
    deterministic_checks: list[DeterministicCheckRef] = Field(default_factory=list)
    rubric: Rubric | None = None

    @model_validator(mode="after")
    def _at_least_one_grader(self):
        if not self.deterministic_checks and self.rubric is None:
            raise ValueError("a case needs deterministic_checks and/or a rubric")
        return self

# --- the contracts a registered check and a run must satisfy ---

class CheckResult(BaseModel):
    name: str
    passed: bool
    evidence: str = ""

# a deterministic check is:  Callable[[EvalContext], CheckResult]
#   where EvalContext carries the produced output + checkout path.

class CaseResult(BaseModel):
    case_id: str
    checks: list[CheckResult]
    rubric_score: float | None = None
    passed: bool                              # overall verdict for the baseline row

M4. EvalCase schema + loader harness

What it is: the Pydantic models above (in eval_def.py) plus a loader from an on-disk case (YAML + referenced check files). Validation: seed_prompt + target_commit required; start_commit optional; at least one of checks/rubric present (enforced by the validator).

MVP done when: a case file loads into EvalCase and round-trips, and a DeterministicCheckRef.ref resolves to a real callable.

M5. Deterministic check runner harness

What it is: runs a case's Python checks against the produced output / checkout and returns structured pass/fail + evidence. Checks are plain Python so authors can write anything (run the repo's tests, grep for a symbol, assert a file exists).

MVP done when: a known-good output passes its checks and a known-bad one fails, with which check failed reported.

M6. Rubric grader harness

What it is: loads a case's rubric and grades the produced output with an LLM judge (each rubric item judged with quoted evidence; total = item aggregate). The judge LLM client is injected so it can be stubbed in tests.

MVP done when: a stubbed judge returns per-item results and an aggregate score for a given output.

M7. Runner harness

What it is: the thing that actually executes a case. Check out start_commit (or a clean base), make the knowledge available (place CLAUDE.md + register the M3 reader tool), run the seed_prompt through headless Claude Code, and capture the produced output (diff / files / text). Then hand the output to M5 + M6 to grade.

MVP done when: given a case, it produces a graded CaseResult. Keep the Claude Code invocation thin; a fake runner backs the harness's own tests so CI stays offline.

M8. Case registry + baseline record harness

What it is: a place to register cases and record their scores so a baseline exists to iterate against. MVP = a directory of cases + a plain JSON/JSONL results file written by a run. No verify, no compare, no gate — just "run the registered cases and write down what they scored."

MVP done when: running the suite emits a results file with one row per case; re-running reproduces comparable numbers.

M9. Worked example case (expected to fail) harness

What it is: one real end-to-end case proving the whole loop runs — a seed prompt, a target commit, deterministic checks, and a rubric. It's fine (expected, even) that the agent doesn't pass it yet; the point is the harness runs and records the failure as a baseline.

MVP done when: the case runs end-to-end through M7 → M5/M6 → M8 and produces a recorded (failing) score. This is the template every future eval is cloned from.

M5 (check runner), M6 (rubric grader), M7 (headless Claude Code runner + fake), and M8 (registry + baseline writer) live within evals/run.py and its helpers for MVP; split into modules when they grow.

Definition of done for the MVP

  • I can call ingest("some insight") and watch CLAUDE.md change.
  • An agent run can call the reader tool and get the whole CLAUDE.md back.
  • I can write a case (5 fields), register it, run it, and get a recorded score — even a failing one.
  • The example case proves the pipeline end-to-end, giving everyone a template to fan out from.

Once that baseline exists, the harness grows into the fuller design in future-work/ (lift, scoreboard, CI gate, real retrieval) without rework.
