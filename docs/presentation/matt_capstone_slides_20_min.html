Describing Capability Accumulation Rigorously

What “more data → more capabilities” means, formally

SymbolMeaning
\( q \)the input you're running now (the “query”)
\( y \)the model's output
\( W \)weight matrix — the model's memory
\( x_i \)training example \(i\)
\( \sigma_i \)error signal for example \(i\)
\( x_i^{\top} q \)how similar the new input is to example \(i\)
Eq 1 — a single-layer network
\[ y = W q \]
Eq 2 — a learning update step
\[ W \;\leftarrow\; W + \sigma_i\, x_i^{\top} \]
Eq 3 — training accumulates those lessons
\[ W_T \;=\; W_0 + \sum_{i=1}^{T} \sigma_i\, x_i^{\top} \]
Eq 4 — inference is a similarity-weighted replay
\[ W_T\, q \;=\; W_0\, q \;+\; \sum_{i=1}^{T} \sigma_i \,\bigl(x_i^{\top} q\bigr) \]

Describing Capability Accumulation Rigorously

The theorem: context selects which capabilities fire

SymbolMeaning
\( q \)the input you're running now (the query)
\( y \)the model's output
\( C \)the context you add
\( F(q) \)attention's output — the steering it adds to \(q\)
\( \tilde q \)the enriched query: \(q + F(q)\)
\( x_i \)capability i's trigger / key — the input that fires it
\( W_{P_0} \)the operator from your original prompt
\( \Delta W_C \)the operator from the added context \(C\)
\( \theta \)a latent skill the model already has
Eq 8 — context biases which capabilities fire
\[ \begin{gathered} \tilde q \;=\; q + F(q) \\[10pt] \underbrace{x_i^{\top}\tilde q}_{\text{capability } i} \;=\; \underbrace{x_i^{\top} q}_{\text{input}} \;+\; \underbrace{x_i^{\top} W_{P_0}\, q}_{\text{prompt}} \;+\; \underbrace{x_i^{\top} \Delta W_C\, q}_{\text{added context}} \end{gathered} \]
Eq 9 — the same statement, one level up
\[ p\,(y \mid q,\, C) \;=\; \sum_{\theta} \; \underbrace{p\,(y \mid q,\, \theta)}_{\text{written by training}} \;\cdot\; \underbrace{p\,(\theta \mid C)}_{\text{selected by context}} \]

Weaknesses of Skills

And the directory itself doesn't scale

  • ✗
    No batching of jobs — skills can't be grouped and run as a set
  • ✗
    No easy batch loading — you can't pull a curated batch of skills for a project
  • ✗
    Per-project rewrite — you re-create the entire skill directory for every project
  • ✗
    No deduplication — overlapping, redundant skills just pile up
  • ✗
    No atomic deletion — you can't cleanly remove a skill and everything that referenced it
  • ✗
    No conflict detection or resolution — contradictory skills sit side by side
  • ✗
    No forced application — you can't require parts of a skill to run, so there's no constructable workflow
  • ✗
    Unreadable soup — the file setup degrades into an unreadable mess
The Solution

A knowledge graph — plus an agent that applies it

🕸A real knowledge graph
  • ✓

  • ✓
    Conflict detection & resolution — contradictions get reconciled
  • ✓
    Partitioned by workflow — the right slice per process
+
🤖An agentic layer
  • ✓
    Finds what must apply to a given workflow
  • ✓
    Forcibly applies it — a constructable, repeatable workflow
